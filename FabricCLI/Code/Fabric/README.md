# Code Preparation Guide — Detailed Reference

This guide covers the full details of preparing Fabric artifacts for deployment. For a quick overview, see the [Deployment README](../../DeploymentScrips/README.md#-preparing-code-for-deployment).

> **Sample Code Available:** The [`SampleCode/`](../../SampleCode/) folder includes ready-to-reference sample **notebooks** and **pipelines** with `##placeholder##` tokens already in place — showing exactly how placeholders are used for lakehouse IDs, workspace references, and notebook bindings.

## Artifact Folder Structure

After exporting, your `Code/Fabric/` directory should look like this:

```
Code/Fabric/
  ├── Notebooks/
  │   └── SampleIngestion.Notebook/
  │       ├── .platform
  │       └── notebook-content.ipynb
  ├── Pipelines/
  │   └── SamplePipeline.DataPipeline/
  │       ├── .platform
  │       └── pipeline-content.json
  ├── Models/
  │   └── SalesAnalysis.SemanticModel/
  │       ├── .platform
  │       └── definition/
  │           ├── model.tmdl
  │           ├── tables/*.tmdl
  │           └── expressions.tmdl
  └── Reports/
      └── SalesAnalysis.Report/
          ├── .platform
          ├── definition.pbir
          └── report.json
```

## Exporting Artifacts

Use `fab export` to pull artifacts from an existing Fabric workspace directly into the right folder:

```bash
# Notebooks
fab export "WorkspaceName.Workspace/MyNotebook.Notebook" -f -o "Code/Fabric/Notebooks/"

# Pipelines
fab export "WorkspaceName.Workspace/MyPipeline.DataPipeline" -f -o "Code/Fabric/Pipelines/"

# Semantic Models
fab export "WorkspaceName.Workspace/MyModel.SemanticModel" -f -o "Code/Fabric/Models/"

# Reports
fab export "WorkspaceName.Workspace/MyReport.Report" -f -o "Code/Fabric/Reports/"
```

Each command creates a folder (e.g., `MyNotebook.Notebook/`) containing a `.platform` file and content files.

## The `.platform` File

Every exported artifact folder contains a `.platform` file. This is auto-generated by `fab export` — you generally don't need to edit it. Here's what it looks like:

```json
{
    "$schema": "https://developer.microsoft.com/json-schemas/fabric/gitIntegration/platformProperties/2.0.0/schema.json",
    "metadata": {
        "type": "Notebook",
        "displayName": "SampleIngestion",
        "description": "Sample notebook for data ingestion"
    },
    "config": {
        "version": "2.0",
        "logicalId": "00000000-0000-0000-0000-000000000000"
    }
}
```

The `type` field matches the artifact kind: `Notebook`, `DataPipeline`, `SemanticModel`, or `Report`.

## Placeholder Reference

Use `##parameterName##` tokens in your code. The deployment scripts replace these with values from `fabric_config.json` at deploy time.

### Where to Find & Replace

| Artifact | File to Edit | What to Replace |
|----------|-------------|-----------------|
| Notebooks | `notebook-content.ipynb` | Lakehouse IDs, storage accounts, workspace IDs |
| Pipelines | `pipeline-content.json` | Notebook references, workspace IDs, lakehouse IDs |
| Models | `*.tmdl` files in `definition/` | Data source connection strings, server names, workspace references |
| Reports | `definition.pbir` | Semantic model ID |

### In Notebooks (`.ipynb`)

```python
# Replace hardcoded GUIDs and names with placeholders
LAKEHOUSE_ID = "##fabricLakehouseId##"
STORAGE_ACCOUNT = "##storageAccountName##"
ENVIRONMENT = "##environmentName##"
WORKSPACE_ID = "##fabricWorkspaceId##"

# These get resolved at deploy time, so your code works across environments
onelake_path = f"abfss://{LAKEHOUSE_ID}@onelake.dfs.fabric.microsoft.com/Tables/mydata"
adls_path = f"abfss://bronze@{STORAGE_ACCOUNT}.dfs.core.windows.net/raw/"
```

### In Pipelines (`pipeline-content.json`)

```json
{
    "typeProperties": {
        "notebookId": "##SampleIngestion##",
        "workspaceId": "##fabricWorkspaceId##",
        "parameters": {
            "lakehouse_id": {
                "value": "##fabricLakehouseId##"
            },
            "environment": {
                "value": "##environmentName##"
            }
        }
    }
}
```

### In Semantic Models (`.tmdl` files)

Look in `definition/expressions.tmdl` or `definition/tables/*.tmdl` for hardcoded connection strings, server URLs, or workspace references. Replace them with placeholders:

```
// Before (hardcoded)
"Server": "powerbi://api.powerbi.com/v1.0/myorg/MyWorkspace"

// After (placeholder)
"Server": "##workspaceConnectionString##"
```

Also check for Log Analytics or other data source references:
```
// Before
"WorkspaceId": "abc12345-xxxx-xxxx-xxxx-xxxxxxxxxxxx"

// After
"WorkspaceId": "##logAnalyticsWorkspaceId##"
```

### In Reports (`definition.pbir`)

```json
{
    "version": "1.0",
    "datasetReference": {
        "byConnection": {
            "pbiModelDatabaseName": "##semanticModelId##",
            "connectionType": "pbiServiceXmlaStyleLive"
        }
    }
}
```

The `##semanticModelId##` is auto-resolved — the scripts deploy the matching model first, capture its GUID, and inject it here.

## Naming Conventions

The deployment scripts use naming conventions to automatically wire artifacts together. Getting these right is critical.

### Reports ↔ Semantic Models

Reports and models **must share the same base name**:

```
✅ CORRECT:
   Model:  LogAnalytics.SemanticModel   → base name: "LogAnalytics"
   Report: LogAnalytics.Report          → base name: "LogAnalytics"
   
❌ INCORRECT:
   Model:  LogAnalytics.SemanticModel
   Report: TelemetryDashboard.Report    → names don't match!
```

**How it works:**
1. Script deploys `LogAnalytics.SemanticModel` → captures its GUID
2. Saves GUID to `modelConfiguration` in `fabric_config.json`
3. When deploying `LogAnalytics.Report`, finds the matching model by base name
4. Replaces `##semanticModelId##` in `definition.pbir` with the GUID

### Pipelines ↔ Notebooks

Pipelines reference notebooks using `##NotebookName##` tokens — **without** the `.Notebook` suffix:

```json
{
    "activities": [
        {
            "name": "Run Ingestion",
            "type": "NotebookActivity",
            "typeProperties": {
                "notebookId": "##DataIngestion##"
            }
        }
    ]
}
```

`DataIngestion.Notebook` → use `##DataIngestion##` ✅  
`DataIngestion.Notebook` → don't use `##DataIngestion.Notebook##` ❌

### Deployment Order

The scripts enforce this order automatically — each step captures IDs for the next:

1. **Semantic Models** → IDs captured for reports
2. **Reports** → bound to models via `##semanticModelId##`
3. **Notebooks** → IDs captured for pipelines
4. **Pipelines** → bound to notebooks via `##NotebookName##`

| `##fabricWorkspaceId##` | Auto-resolved from workspace name | `xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx` |

| `##tenantName##` | From `fabric_config.json` | `contoso` |
| `##logAnalyticsWorkspaceId##` | From `fabric_config.json` | `xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx` |
| `##semanticModelId##` | Auto-resolved per model | GUID of deployed model |
| `##NotebookName##` | Auto-resolved per notebook | GUID of deployed notebook |
